# Leak Data Reader PoC — System Architecture

## Overview

This codebase is a **Proof of Concept for a generic Leak Data Reader** — a full-stack
OSINT tool for ingesting, enriching, and investigating publicly available leaked email
datasets. It is structured as a **pnpm monorepo** with three runtime artifacts and four
shared library packages.

> **Proof of Concept:** This tool demonstrates a reusable framework for email-leak
> forensics. It ships with sample data from [wikiran.org](https://wikiran.org) as an
> illustrative dataset. The ingestion pipeline, entity extraction, and identity clustering
> are all source-agnostic and can be pointed at any publicly available leaked email corpus.

> **Dataset selection note:** wikiran.org was chosen as the example data source because
> its leaked email archives are publicly available and do not contain information that
> would be considered classified or sensitive in Western or English-speaking countries.
> This tool is not intended to target Iran or Iranian individuals. The framework is
> source-agnostic and can be applied to any publicly available leaked email corpus
> regardless of national origin.

```
workspace root/
├── artifacts/
│   ├── api-server/          Node.js + Express REST API (port from $PORT)
│   ├── wikiran/             React + Vite frontend (port from $PORT)

├── lib/

│   └── api-zod/             Generated Zod request/response validators
└── scripts/                 One-off seed and maintenance scripts
```

## Services and Ports

Each artifact reads `process.env.PORT` (or `$PORT` in Vite). The Replit proxy routes traffic
based on path prefix — there is no hardcoded port in any application code.

| Artifact | Path prefix | Technology |
|---|---|---|
| `wikiran` | `/` | React 19, Vite 7, TanStack Query 5, shadcn/ui, Tailwind 4 |
| `api-server` | `/api-server` (proxy) | Express 5, Drizzle ORM, OpenAI SDK |
| `mockup-sandbox` | `/mockup-sandbox` | Vite dev server (dev only) |

The frontend constructs API URLs using `import.meta.env.BASE_URL` (Vite's base path),
stripping the trailing slash: `API_BASE = import.meta.env.BASE_URL.replace(/\/$/, "")`.
All API calls use the path `${API_BASE}/api/<endpoint>`. There is no separate `VITE_API_URL`.

## Shared Packages

### `@workspace/db` (`lib/db/`)

Single source of truth for the database schema. Exports:

- Drizzle table definitions (`datasetsTable`, `emailsTable`, …)
- Drizzle insert/select TypeScript types
- Zod insert schemas (via `drizzle-zod`)
- A singleton `db` client connected to `DATABASE_URL`

Both `api-server` and any future services import from `@workspace/db`. Route handlers use
Drizzle's `sql` tagged-template literal (imported from `drizzle-orm`) for complex
conditions (e.g. `ANY(...)`, `ILIKE`, subqueries) directly inside the route files — this
is the Drizzle-idiomatic approach and does not bypass the ORM.

### `@workspace/api-client-react` (`lib/api-client-react/`)

Generated by [orval](https://orval.dev/) from `lib/api-spec/openapi.yaml`. Provides one
TanStack Query hook per API endpoint (e.g. `useListEmails`, `useGetThread`,
`useCreateIdentity`). These hooks cover most API calls, but several pages use raw `fetch`
directly — notably `crypto.tsx` (all crypto API calls) and `emails.tsx` (attachment
OCR and OCR translate operations), and `use-auth.tsx` (all auth operations).

To regenerate after an OpenAPI spec change:
```bash
pnpm --filter @workspace/api-client-react run generate
```

### `@workspace/api-zod` (`lib/api-zod/`)

Also generated by orval. Exports Zod schemas for every request body and response shape.
Currently used for optional server-side validation; the API routes do not yet enforce these
schemas at runtime (see `docs/EXTENDING.md` for the pattern to add validation).

## Data Flow: Browser → Database

```
Browser
  │
  │  HTTP (TanStack Query hook)
  ▼
api-server  (/api/*)
  │
  ├─ Auth middleware (JWT verify on Authorization header)
  │
  ├─ Route handler (routes/*.ts)
  │     │
  │     ├─ Drizzle ORM query  →  PostgreSQL (DATABASE_URL)
  │     │
  │     └─ AI call (optional)  →  OpenAI proxy (AI_INTEGRATIONS_OPENAI_BASE_URL)
  │
  └─ JSON response
```

### Request lifecycle detail

1. The React frontend calls a generated hook such as `useGetEmails({ dataset_id: 3 })`,
   or in the case of auth operations, calls `fetch` directly from `use-auth.tsx`.
2. For generated-hook calls, TanStack Query fires a request via `customFetch`. For
   auth operations, `use-auth.tsx` manually adds `Authorization: Bearer <token>`.
   **Note:** The generated `customFetch` wrapper does not inject auth headers automatically.
3. Express routes the request to the appropriate router.
4. The route handler builds a Drizzle query and awaits the PostgreSQL result.
   Only `auth/change-password` and `auth/me` verify the JWT; all other handlers
   process requests without authentication checks.
5. If an AI call is needed (translation, thread analysis, entity extraction), the handler
   calls a function from `lib/ai.ts`, which POSTs to the OpenAI-compatible proxy endpoint
   configured via environment variables.
6. The handler serializes the result to JSON and returns it.
7. TanStack Query caches the result and re-renders the component.

## Auth Layer

Authentication is **JWT-based**, implemented in `routes/auth.ts` and `hooks/use-auth.tsx`.
There is no session store or cookie.

### Token lifecycle

| Step | Detail |
|---|---|
| Login | `POST /api/auth/login` → bcrypt compare → sign 7-day JWT with `{ sub, username, mustChangePassword }` |
| Storage | JWT is written to **`localStorage`** under the key `"wikiran_token"` and mirrored in React state. On page reload, `useAuth` reads from `localStorage` and re-validates via `GET /api/auth/me`. |
| Outgoing requests (auth) | `use-auth.tsx` calls `fetch` directly and manually sets `Authorization: Bearer <token>` |
| Outgoing requests (API) | The generated hooks in `@workspace/api-client-react` use a `customFetch` wrapper that does **not** automatically inject auth headers. Pages that need auth must pass the token via the hook's `request` options or rely on backend routes that do not enforce authentication. |
| Backend enforcement | Only `POST /api/auth/change-password` and `GET /api/auth/me` call `jwt.verify` on the incoming token. All other routes (emails, threads, entities, identities, etc.) perform **no server-side JWT verification**. Access control is enforced at the frontend level via `AuthGate`. |
| Password change | `POST /api/auth/change-password` → re-hashes password → issues a fresh token with `mustChangePassword: false` |

**Security posture summary:** This API is not protected at the HTTP level except
for the two `auth/` endpoints that verify the JWT. Any client with network access to the
API server can call any data endpoint without a token. The login wall is a frontend-only
guard. This is intentional for the current single-tenant, closed-network deployment but
should be hardened before any public or multi-tenant deployment.

**`JWT_SECRET`** must be set in production via the `JWT_SECRET` environment variable.
The fallback string `"wikiran-dev-secret-do-not-use-in-prod"` is only safe in local dev.

### First-login flow

All users created via the seed script have `mustChangePassword = true`. After login, the
frontend's `AuthGate` detects this flag and renders `<ChangePasswordModal>` rather than
the main app. The user cannot access any page until the password is changed.

## AI Integration

AI calls use the Replit AI Integrations proxy — a drop-in replacement for the OpenAI API
that requires no personal API key. Two environment variables configure it:

```
AI_INTEGRATIONS_OPENAI_BASE_URL   # proxy base URL
AI_INTEGRATIONS_OPENAI_API_KEY    # proxy API key (set by Replit integration)
```

The OpenAI client is instantiated in `artifacts/api-server/src/lib/ai.ts`:

```typescript
const openai = new OpenAI({
  baseURL: process.env.AI_INTEGRATIONS_OPENAI_BASE_URL,
  apiKey: process.env.AI_INTEGRATIONS_OPENAI_API_KEY ?? "placeholder",
});
```

Route handlers call AI functions directly — there is no queue or background worker. If an
AI call fails, the error propagates to the route handler and returns a 500 to the client.

**Where AI functions live:**

Most AI functions are in `artifacts/api-server/src/lib/ai.ts`. However, the attachment
OCR logic has its own inline OpenAI client instantiated directly in
`artifacts/api-server/src/routes/attachments.ts` — it was not extracted to `lib/ai.ts`.

**Models in use:**

| Function | File | Model | Purpose |
|---|---|---|---|
| `translateEmailBody` | `lib/ai.ts` | gpt-5-mini | Translate email body to English |
| `translateOcrText` | `lib/ai.ts` | gpt-5-mini | Translate OCR-extracted attachment text |
| `translateEntities` | `lib/ai.ts` | gpt-5-mini | Romanize/translate entity names |
| `analyzeThread` | `lib/ai.ts` | gpt-4o | Generate thread summary + investigative leads |
| `extractEntitiesFromText` | `lib/ai.ts` | gpt-5-mini | NER on email body text |
| `extractIdentityClusters` | `lib/ai.ts` | gpt-4o | Cluster individuals from email metadata |
| Attachment OCR (inline) | `routes/attachments.ts` | gpt-4o | PDF text cleanup + image OCR via vision |

## Environment Variables

| Variable | Service | Required | Purpose |
|---|---|---|---|
| `DATABASE_URL` | api-server | Yes | PostgreSQL connection string |
| `JWT_SECRET` | api-server | Prod only | JWT signing secret |
| `AI_INTEGRATIONS_OPENAI_BASE_URL` | api-server | Yes | AI proxy base URL |
| `AI_INTEGRATIONS_OPENAI_API_KEY` | api-server | Yes | AI proxy key |
| `PORT` | both | Yes | HTTP listen port (set by Replit) |
| `BASE_URL` | wikiran | Yes | Vite base path (set by Replit); frontend derives API base from this |
